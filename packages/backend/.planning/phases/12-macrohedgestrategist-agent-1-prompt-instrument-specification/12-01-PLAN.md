## Two design forks — RESOLVED for this phase (document in the SUMMARY)

**Fork B — `IMacroThesis` registry shape: HANDLE-RESOLVING (chosen over enum-indexed).**
Rationale (RESEARCH §"IMacroThesis registry" + Pitfall 5): the mandate's `economicTheory` resolves to an `IMacroThesis` handle that drops straight into `HedgeLegParams.economicTheory` with ZERO translation — no hidden enum->handle seam at the Phase-14 hand-off. For the demo the handle is a deterministic non-zero SENTINEL per school (the UI renders the human label from the event string; "address is opaque" per UI model). The registry is the single source of truth for both the `allowedValues` guardrail and the `_mapSchool` keccak compare.

NON-DEPLOYABLE SENTINELS: the demo handles (`address(uint160(0x5))` / `0x6`) are placeholder sentinel addresses, NOT deployed contracts. They satisfy ONLY the `!= address(0)` well-formedness check — NEVER `call`/`delegatecall`/`staticcall` them (no code lives there). A school needs no on-chain behavior this phase; the load-bearing property is solely "resolved handle is non-zero".

(The prompt-provenance fork A/B — caller free-text intent vs deterministic macro prompt — is a Plan-02 concern; this plan only ships the type + registry.)
</decisions>

    /// @notice The concrete named-thesis registry (STRAT-01). Single source of truth for BOTH the

    library MacroThesisRegistry {
        // (a) the allowedValues for the inferString school leg — the STRUCTURAL guardrail.
        function schoolLabels() internal pure returns (string[] memory labels) {
            labels = new string[](2);          // extensible: add a label here + a branch in thesisOf/promptBias

        function thesisOf(string memory label) internal pure returns (IMacroThesis handle, bool ok) {

            return "";

    }
