The contract reads a `MacroOracle` datum, builds a deterministic macro-hedging prompt over `(actual, consensus)`, and fires TWO LLM-Inference requests across TWO transactions — `requestActionDecision` fires `inferString` (action enum, constrained to `allowedValues`) and `requestSizeDecision` fires `inferNumber` (size, bounded `[0, MAX_SIZE_BPS]`) — ONE per transaction (the whole-msg.value forwarding footgun).

CRITICAL JOIN DESIGN (the BLOCKER-1 fix; supersedes the PRD's single `requestHedgeDecision` — see 11-CONTEXT.md DIVERGENCE note): the two legs run in DIFFERENT blocks (the keeper sequences action → await callback → size). The `decisionId` MUST therefore be STABLE across the two transactions and collision-free — derive it from the action request's own `requestId` (monotonic via MockPlatform/live platform) or a contract-incremented nonce, NEVER `keccak(...block.number)` (which would compute a different id per leg → `actionSet && sizeSet` never both true on one struct → `HedgeDecisionMade` never fires on testnet). `requestActionDecision` RETURNS/EMITS the `decisionId`; `requestSizeDecision(bytes32 decisionId)` takes that SAME id EXPLICITLY and binds the size leg to the already-allocated struct.

The callback decodes via the inherited authenticated `handleResponse` -> `_onResult` (NEVER a bespoke `require(msg.sender==PLATFORM)`), maps the action string to a `HedgeAction` enum behind a try/catch (so a malformed/wrong-type payload routes to `DecisionFailed` and leaves `actionSet==false` — no enum-zero write — instead of bricking the pending request), clamps the size (negative → 0, over-max → MAX_SIZE_BPS), and once BOTH legs have landed on the same `decisionId`, stores the completed `HedgeDecision` and emits `HedgeDecisionMade`. The `allowedValues`/`[min,max]` bounds ARE the structural guardrail — full autonomy stays in-bounds.
