<objective>
Implement the two ROADMAP-named fuzz invariants whose `.tree` was authored in Plan 01 (`LongGammaWrapper.invariants.tree`): `invariant_residualNeverExceedsHoldings` (a SINGLE claimResidual payout never exceeds the live `convertToAssets(balanceOf(wrapper))` snapshotted just before THAT redeem) and `invariant_userClaimsBackedByCollateral` (the CUMULATIVE payouts never exceed the running sum of the live surviving collateral the wrapper actually HELD at each claim time — a non-overpayment upper bound, NOT the gross deposit). A bounded Foundry invariant handler drives the lifecycle (deposit -> swap -> syncResidual -> close -> claimResidual) and the invariants hold across all fuzz runs. This is the phase's structural-soundness gate.

Purpose: These are the two named invariants the ROADMAP/STATE process gate requires at a fixed CI fuzz-run floor. They prove the foundational safety property — the wrapper is never on the hook for more than it holds (already structurally enforced by `settleBurn`'s `NotEnoughTokens`, now fuzz-asserted) and the wrapper never PAYS OUT, cumulatively, more than the live collateral it held at the moments it actually paid (no over-promise, no double-pay, never paying from the gross deposit figure). The two invariants are DISTINCT: `residualNeverExceedsHoldings` bounds a single claim's payout; `userClaimsBackedByCollateral` bounds the CUMULATIVE payout stream against summed live holdings. Both are ROADMAP-named, so neither may be deleted — both must be true AND non-vacuous.
Output: `LongGammaWrapper.invariants.t.sol` (the tree pre-exists) green at the fuzz floor.

CRITICAL — WHY "promised = the gross deposit" IS WRONG (B1, source-verified): minting the long burns a commission IMMEDIATELY (CollateralTracker.sol L1552-1559: `commissionFee = mulDivRoundingUp(longAmount+shortAmount, notionalFee(), DECIMALS)`, then `_burn(optionOwner, sharesToBurn)`). So right after the first `act_open`, `convertToAssets(balanceOf(wrapper)) = deposit − commissionFee < deposit` by FAR more than a `+1` tolerance — at the handler's low deposit bound (`a0 ∈ [1e6, 1e14]`) the gap is large. Therefore the wrapper does NOT (and per WRAP-03/04 MUST NOT) retain the gross deposited notional as backing; residual is read FROM surviving, never the deposit. Any invariant of the form `liveSurv + totalPaid + 1 >= deposit` is INVERTED — it false-reds a correct wrapper on fuzz step 1. The backing invariant must instead be an UPPER-BOUND on realized payouts vs the live holdings the wrapper actually had at claim time.

INVARIANT SEMANTICS (08-RESEARCH Validation Architecture + ROADMAP) — assert an INDEPENDENT handler ledger vs LIVE wrapper state, never re-derive owed from surviving, never reference the gross deposit as backing:
  - invariant_residualNeverExceedsHoldings (bounds a SINGLE claim): the handler captures `preClaimSurviving_i = convertToAssets(balanceOf(wrapper))` IMMEDIATELY BEFORE each `claimResidual()` and the realized `lastPaid_i` from the ResidualClaimed event (or the user balance delta). The invariant asserts `handler.lastPaid_i <= handler.preClaimSurviving_i + 1` — an independently-recorded payout vs an independently-snapshotted live holding. NOT a tautology: `paid` comes from the redeem path (maxRedeem-capped), `preClaimSurviving` from convertToAssets — two distinct reads the redeem logic must keep ordered.
  - invariant_userClaimsBackedByCollateral (bounds the CUMULATIVE payout stream — the non-overpayment / upper-bound property): the handler maintains TWO independent accumulators, BOTH driven from action side-effects, NEVER from the deposit:
      • `totalPaid_i` += `lastPaid_i` at each claim (the realized cumulative payout to the user=handler).
      • `cumPreClaimSurviving_i` += `preClaimSurviving_i` at each claim (the running SUM of the live `convertToAssets(balanceOf(wrapper))` the wrapper actually HELD at the instant of each redeem, snapshotted BEFORE the redeem).
    The invariant asserts `handler.totalPaid_i <= handler.cumPreClaimSurviving_i + 1` — "the wrapper never pays the user, cumulatively, more than the collateral it actually held at the times it paid." This is:
      - TRUE on fuzz step 1 (post-mint): before any claim, `totalPaid_i == 0` and `cumPreClaimSurviving_i == 0`, so `0 <= 0 + 1`. It NEVER references the gross deposit, so the commission burn cannot false-red it.
      - NON-VACUOUS: it compares realized payouts (redeem path) to an independently-snapshotted live-holdings sum. It would FAIL a wrapper that pays from the deposit figure (payout > the live surviving it held) or double-pays (a second payout with no second pre-claim snapshot, so totalPaid outruns cumPreClaimSurviving). It is NOT `max(surv-cost,0)<=surv`.
  NOTE: do NOT write `owed = max(surv-cost,0); assertLe(owed, surv)` — true by construction, proves nothing (W4). Do NOT write `liveSurv + totalPaid + 1 >= deposit` / `>= promised` / `>= a0` — INVERTED (commission burn makes liveSurv < deposit on step 1). The real fund-safety guarantee is the `settleBurn` `NotEnoughTokens` revert; the fuzz invariants add value by checking INDEPENDENT realized-payout ledgers against LIVE / summed-live holdings, never against the deposit.
