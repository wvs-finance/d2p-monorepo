# Phase 05.2: DEFI-06 wallet connect-modal accessibility — Context

**Gathered:** 2026-06-02
**Status:** Ready for planning
**Source:** Direct code investigation (orchestrator) — DEFI-06 acceptance criteria are the spec.

<domain>
## Phase Boundary

Close the last open Phase-5 requirement, **DEFI-06**: "Wallet connect modal is fully
keyboard-navigable, has no focus trap on close, and announces state changes to screen readers."

Scope: the wallet **connect-modal** a11y flow (the RainbowKit `<ConnectButton>` modal opened from
`WalletPanel`), plus the SR-announcement of wallet state transitions. Verify + harden + test +
Accessibility-Auditor sign-off. Frontend-only; read-first (no transact).
</domain>

<decisions>
## Findings (current state — verified by code read)

- `components/defi/WalletPanel.tsx` already wraps its render in `aria-live="polite" aria-atomic="true"`
  (the SR announcement boundary) and renders RainbowKit's `<ConnectButton>` in the DISCONNECTED
  (non-readOnly) state. RainbowKit's modal owns its own focus trap (react-remove-scroll).
- **The connect modal has NO reachable surface today:** `<ConnectButton>` renders only on the
  non-readOnly path (`page.tsx:377`), but the only registry entry is the **simulated (readOnly)**
  instrument — so the modal never appears on any reachable route.
- Existing "DEFI-06" tests (`tests/a11y/defi-instruments.spec.ts`, `tests/e2e/wallet-states.spec.ts`)
  cover **axe AA on the pages + the aria-live region presence on the readOnly path** — they do NOT
  exercise the modal's keyboard nav, focus restoration on close, or connect-flow SR announcements.

## Implementation Decisions (LOCKED)

- **Verification surface = a temporary scratch route** (precedent: the `_modal-check` scratch route
  used in 05-02, commits `f6a1419` / `c82c653`). Render a non-readOnly `WalletPanel` (DISCONNECTED →
  `<ConnectButton>`) at a throwaway route, exercise the modal for keyboard/focus/SR + Accessibility
  Auditor, then **remove the route** in the same phase. No permanent connect surface is added (v1 is
  read-first with no live instrument — a production connect affordance is out of scope).
- **Harden** whatever the Accessibility Auditor finds: at minimum verify (a) keyboard reaches and
  operates the modal, (b) focus returns to the `<ConnectButton>` trigger on close (no focus loss /
  no trap-after-close), (c) wallet state transitions (DISCONNECTED→CONNECTING→…) announce via the
  existing `aria-live` region. The `useMounted` guard from 05.1-02 already suppresses the CONNECTING
  flash — confirm it doesn't suppress legitimate announcements.
- **Test coverage (permanent):** add e2e assertions for keyboard-open + focus-return-on-close +
  aria-live announcement, written so they do NOT depend on the (removed) scratch route where
  possible (prefer a component/integration render of `WalletPanel` non-readOnly; use the scratch
  route only for the full RainbowKit-modal e2e, and gate it behind a check that tolerates its
  removal — or keep the modal e2e as a documented manual/Accessibility-Auditor verification if the
  scratch route can't persist).
- **Accessibility Auditor sign-off** (keyboard + screen-reader + axe against DEFI-06's three
  criteria) is the verification gate — recorded in the phase LIVE-VERIFICATION.
- es-CO-first for any new copy; locked tokens; no `--no-verify`.
