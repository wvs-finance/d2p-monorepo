<task number="1" type="auto">
<action>
**Fix the aria-live mechanism + focus-return (the two BLOCKERs).**

1. `WalletPanel.tsx`: REMOVE `aria-live="polite" aria-atomic="true"` from the outer wrapping `<div>`.
   Add a DEDICATED announcement node — a visually-hidden `<p role="status" aria-live="polite">` (use
   the existing `sr-only` class) that renders ONLY the current state's label text
   (`strings.statusLabels[walletState]`). Visible UI stays as-is, outside the live node. This makes
   transitions announce just the label, not a panel-wide re-read.
2. **Focus-return / connect-success:** capture a ref to the connect trigger. Because RainbowKit's
   bare `<ConnectButton>` gives no trigger handle and the DISCONNECTED branch UNMOUNTS on connect
   success (focus would fall to `<body>`), when `walletState` transitions AWAY from DISCONNECTED
   move focus to the status node: give the `role=status` node `tabIndex={-1}` and `.focus()` it in a
   `useEffect` keyed on `walletState` leaving 'DISCONNECTED'/'CONNECTING'. Do NOT add a second modal
   focus trap (RainbowKit owns the in-modal trap). For Escape/X/backdrop close WITHOUT connecting,
   verify in Task 3 that RainbowKit restores focus to its own trigger; only add a shim if the audit
   finds focus lost.
3. `WalletStatusPill.tsx`: leave the `useMounted` guard, but ensure the FIRST real transition after
   mount is reflected (the status node in WalletPanel reads `walletState`, which is correct post-mount).
4. `tests/unit/wallet-panel-connect-a11y.test.tsx` (NEW, jsdom; mock `useAccount`, and mock
   `ConnectButton: () => <button>connect</button>` per the existing wallet-read-only.test pattern):
   assert (a) a `[role="status"][aria-live="polite"]` node exists and contains the current status
   label; (b) the outer wrapper no longer has `aria-atomic`; (c) re-render disconnected→connecting and
   assert the status node's text updates. Do NOT assert RainbowKit's real accessible name here (that's
   the live audit's job — jsdom mock can't prove it).
</action>
