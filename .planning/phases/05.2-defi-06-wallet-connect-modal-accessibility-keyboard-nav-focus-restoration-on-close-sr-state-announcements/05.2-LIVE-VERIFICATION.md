## Issues found (‚â•3)

### Issue #1 ‚Äî Spanish status labels inherit `lang="en"` (no `lang="es"`) ‚Äî SERIOUS
**WCAG 3.1.2 Language of Parts (AA)** ¬∑ also 3.1.1 Language of Page.
The `<output>` status text and panel copy are es-CO ("Desconectado"/"Conectando"/"Conectado", "Conecta tu billetera‚Ä¶") but the only `lang` in scope is `html[lang=en]`. SRs will pronounce Spanish with an English voice. **User impact:** Spanish-speaking SR users hear garbled, mispronounced status ‚Äî directly undermines the es-CO-first copy mandate.
**Fix:** Add `lang="es-CO"` (or `lang="es"`) to the panel root, or to the `<output>` and prompt nodes. For the production locale-switched app the `<html lang>` should already reflect the active locale ‚Äî verify the real app sets `lang="es"` when es-CO is active; the audit shell does not.

### Issue #2 ‚Äî Mock Connector does NOT appear in the RainbowKit modal ‚Äî SERIOUS (blocks the spec'd UI test path)
The task spec states "a 'Mock Connector' option appears in the RainbowKit modal." It does **not**. The modal's wallet-list pane renders **empty** (no `rk-wallet-option-*` nodes; no "Mock" text anywhere in the DOM); only the "¬øQu√© es una billetera?" intro pane shows. Root cause: the wagmi `mock()` connector is passed to `createConfig.connectors`, but RainbowKit builds its visible list from its own wallet registry (`connectorsForWallets`/`wallets`), not raw wagmi connectors. **User/test impact:** the connect-success flow cannot be driven *through the modal UI* ‚Äî the audit surface's stated purpose ("e2e tests can drive a real DISCONNECTED‚ÜíCONNECTING‚ÜíCONNECTED transition" via a modal click) is unmet. (Verification proceeded by driving the wagmi connect programmatically; the focus/SR logic itself is correct ‚Äî see criteria 2 & 3.)
**Fix:** register the mock connector through RainbowKit's wallet API so it surfaces as a modal option, OR document that the connect transition is driven outside the modal and adjust the e2e to match.

### Issue #3 ‚Äî `<output>` relies solely on implicit `role=status` / `aria-live` ‚Äî MINOR
The live region sets no explicit `role="status"` or `aria-live="polite"`. The implicit `<output>` mapping is correct and well-supported, but is the kind of implicit-semantics dependency that silently breaks if the element type ever changes. Defense-in-depth: add explicit `role="status"` (and optionally `aria-live="polite"`). Low impact today.

### Issue #4 ‚Äî Trigger focus ring is UA-default blue, not the d2p ochre token ‚Äî MINOR
Visible (2.4.7 passes) but `rgb(11,87,208)` ignores the locked ochre accent. RainbowKit ConnectButton doesn't inherit the design-system focus ring. Cosmetic / brand-consistency.

### Note ‚Äî RainbowKit accent contrast 3.44:1 is a d2p config choice, not purely vendor ‚Äî MINOR
The waived `color-contrast` failure uses `accentColor #a87c3a` + `accentColorForeground #f8f5f0` set in `AuditShell.tsx`. Although rendered by RainbowKit, the failing colors are chosen by d2p. If the real app reuses these RainbowKit theme values, the 3.44:1 button-label contrast is a genuine 1.4.3 failure, not just a vendor waiver. Verify the production RainbowKit theme.

### Non-issue (test artifact, NOT a defect) ‚Äî "Maximum update depth exceeded"
A React infinite-update error appeared **only after** the programmatic `cfg.setState(...)` connect manipulation (conflicting with RainbowKit/AppKit's own store subscription). On a clean fresh load the console shows only the two expected WalletConnect dev network errors (403/400 from the placeholder projectId). The render loop is a harness artifact of forcing wagmi state via the fiber; it is **not** attributable to the DEFI-06 components.

### Issue #2 ‚Äî Mock connector not in RainbowKit modal (SERIOUS) ‚Äî RESOLVED BY SPEC AMENDMENT

The mock connector is NOT surfaced in the RainbowKit modal (wagmi `mock()` uses wagmi's connector
registry; RainbowKit uses its own `connectorsForWallets` registry ‚Äî the two are independent). Rather
than attempting a complex RainbowKit-registry integration, the spec was amended per the audit's

Code comment in AuditShell.tsx and e2e spec explain the architecture. ‚úì

### Issue #3 ‚Äî Implicit role=status (MINOR) ‚Äî WAIVERS RECORDED

Biome v1.9.4 rules `noRedundantRoles` and `useSemanticElements` correctly flag `<output role="status">` as redundant (the `<output>` element already carries implicit `role=status` per HTML-AAM). Adding an explicit `role` attribute triggers a lint error that cannot be suppressed via JSX `{/* biome-ignore */}` comments for these specific rules. Defense-in-depth is provided by the **explicit `aria-live="polite"` attribute** (not all AT stacks apply the implicit live region from `<output>`). The e2e test asserts `aria-live="polite"` and verifies `role` is null (implicit only). WAIVER recorded: explicit `role="status"` blocked by Biome gate; implicit role + explicit aria-live is the pragmatic solution.

### Issue #4 ‚Äî Trigger focus ring (MINOR) ‚Äî ATTEMPTED + WAIVERS RECORDED

`[data-rk] button:focus-visible` ochre override added to `app/globals.css`. Whether RainbowKit's
vanilla-extract cascade overrides this at runtime depends on specificity at the actual rendered node.
If the ochre ring does not appear in practice, the UA-default blue ring still passes WCAG 2.4.7 ‚Äî
brand consistency issue only, not accessibility failure. WAIVER-RBK-FOCUS recorded in globals.css
alongside WAIVER-05-03 (accentColor contrast).
