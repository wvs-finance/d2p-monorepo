## Summary

Phase 05.1 is a frontend-only extension of the existing Phase-5 instrument-detail route. The technical work divides into six discrete problem areas, all of which can be resolved from existing source code without any external API or library research.

**The 05-04 live-verification BLOCKER is a precondition**: the PayoffDiagram currently renders 0-height at all viewports because the parent `<div>` uses `min-h-*` (which does not establish a resolved height for `height:100%` children) instead of `h-*`. The fix is mechanical — changing `min-h-[240px]` to `h-[240px]` on the wrapper div in `PayoffDiagram.tsx`. The contrast failure (3.44:1 on the "Conectar billetera" CTA) is in RainbowKit's `ConnectButton`, not in `PayoffDiagram`; the payoff curve stroke token (`--accent-default: oklch(0.6 0.08 70)`) resolves to `#a87c3a` which is 3.44:1 on `--bg-canvas` — sub-AA for small text. The diagram curve is 2px stroked SVG on a light background — using `--text-primary` (`oklch(0.2 0.01 80)` ≈ 9:1) or `--accent-text` (`oklch(0.48 0.08 70)` which clears 4.5:1) for the SVG stroke resolves this. The React #418 hydration mismatch is caused by `ssr:false` in `PayoffDiagramClient` — the component correctly skips SSR but the loading skeleton div uses `min-h-*` on SSR output vs nothing on client hydration; using the same `h-*` class on the loading skeleton eliminates the mismatch.

**The discriminated-union migration** is mechanical TypeScript work with a known blast radius: three consumer files (`page.tsx`, `InstrumentParams.tsx`, `InstrumentJsonLd.tsx`) each access `strike`/`slope`/`address`/`deployedAt` without narrowing. All three patterns are straightforward `if (instrument.kind === 'live') { ... }` guards. The RSC→client serialization boundary (PayoffDiagramClient) already accepts plain serializable props (`strike: number`, `slope: number`) so switching to `data: PayoffPoint[]` is a safe widening — the spec mandates this for the schematic convex path.

**The GitBook/Velite interaction** requires confirming that `docs/book/` does not match the Velite glob `content/research/*.{es,en}.mdx`. The current Velite config uses `root: 'content'` and patterns `research/*.{es,en}.mdx` — a `docs/book/` directory is entirely outside the `content/` root and will not be touched. The `.gitbook.yaml` at repo root is a static config file that Next.js and Velite both ignore.

**Primary recommendation:** Fix the 05-04 BLOCKERs first (a single commit on `PayoffDiagram.tsx`: `h-*` parent + stroke token swap + loading skeleton class parity). Then execute the four-wave order from the spec: data layer → components → page branch → tests + Evidence Collector.

### Pattern 1: Recharts convex curve via `type="monotone"` + external `PayoffPoint[]`
**What:** Pass pre-computed `PayoffPoint[]` as a prop instead of computing inside `PayoffDiagram`. Add `type="monotone"` to the `<Line>` element. Increase point density near the curvature peak for visual smoothness.

**Why this works:** `type="monotone"` uses the Monotone Cubic spline interpolator (Fritsch-Carlson) which preserves monotonicity — it will not create artificial wiggles in the convex region. With 100+ points, the convex shape reads smooth even at mobile viewports. The current `type="linear"` (piecewise-linear) is fine for the existing kink payoff but would look like a polygon for a smooth convex shape.

### Pattern 2: Schematic convex payoff generator
**What:** Add `generateSchematicConvexPayoff` to `payoff.ts`. Returns `PayoffPoint[]` with a smooth convex (positive-gamma) shape — payoff curves upward as price moves away from center in either direction, net of a fixed premium.

**Formula (Claude's discretion — must be convex + premium-shifted + labeled schematic):**
```typescript
// Source: payoff.ts — extend alongside existing generatePayoffData
// Simple smooth convex: payoff(S) = max(k * (S - S0)^2 - premium, 0)
// where k controls curvature, S0 is the center price, premium is the cost shift.
// This is positive-gamma (∂²payoff/∂S² = 2k > 0) everywhere.
// Points evenly spaced from 0.3*S0 to 1.7*S0 (same domain as the existing generator).
export function generateSchematicConvexPayoff(
  centerPrice: number,
  premium: number,
  curvature = 0.000005,  // discretion; chosen so peak payoff is visually significant
  points = 200,           // dense sampling for smooth visual
): PayoffPoint[]
```

**Unit test contract:**
- All points must have `payoff >= 0` (floor at zero after premium subtraction).
- At least one point must have `payoff > 0` (non-trivial curve).
- The function must be monotone-convex: for prices above the minimum-payoff point, payoff must increase as price moves further from center.
- `generateSchematicConvexPayoff` must NOT call `generatePayoffData` — it is a separate, independent generator.

### Pattern 4: Read-only wallet path
**What:** Add a `readOnly` prop to `WalletPanel`. When `readOnly` is true, render a dedicated read-only state pill and body without calling `deriveWalletState`.

### Anti-Patterns to Avoid

- **Using `min-h-*` as the parent of `ResponsiveContainer height="100%"`**: this is the exact 05-04 BLOCKER. Always use `h-*`.
- **Reading `instrument.strike` / `instrument.slope` / `instrument.address` without narrowing on `kind` first**: will cause TypeScript errors once the union is in place. Every access site must narrow.
- **Passing a `simulated` instrument to `aggregateAllChains` or `getInstrumentPoolState`**: these functions expect a deployed contract address; a simulated entry has no address. The page must branch on `kind === 'simulated'` before any aggregator call.
- **Using `--accent-default` for SVG curve stroke**: resolves to `oklch(0.6 0.08 70)` = 3.44:1 contrast on `--bg-canvas` — sub-AA. Use `--accent-text` (`oklch(0.48 0.08 70)`) instead.
- **Fabricating numbers in `InstrumentJsonLd` for the simulated variant**: the spec is explicit — no `strike`, `slope`, or `address` in the JSON-LD for simulated instruments. Agents consuming the JSON-LD must never receive fabricated financial data.
- **Emitting 0 for unknown values**: always em-dash (`—`). The anti-fishing discipline treats `0` and a real zero as indistinguishable to a downstream consumer.
- **Adding a `deriveWalletState` path that returns `'READ_ONLY'`**: the deriver is pure and unit-tested. The `readOnly` prop short-circuits the deriver in `WalletPanel` before calling it — the deriver never knows about `readOnly`.
- **Card-in-card layout for `CashFlowWaterfall`**: CROSS-04 / `impeccable detect` will catch it. Use a `<dl>` directly on the section background.

## Common Pitfalls

### Pitfall 1: The 0-height chart (05-04 BLOCKER)
**What goes wrong:** `PayoffDiagram` renders an empty whitespace gap; no curve visible.
**Why it happens:** `<ResponsiveContainer height="100%">` resolves height against its parent. If the parent uses `min-h-*`, CSS layout computes parent `height: auto` (content-driven) → 0px because the only content is the zero-size responsive container. `min-h-*` applies a minimum after layout, too late.
**How to avoid:** Parent must use `h-[240px] sm:h-[320px]` (explicit height), NOT `min-h-*`.
**Warning signs:** `.recharts-responsive-container` height = 0px; recharts console warning "The width(0) and height(0) of chart should be greater than 0".
**Current status:** The working-tree `PayoffDiagram.tsx` already has `h-[240px]` on line 62 per the code read. The loading skeleton in `PayoffDiagramClient.tsx` still uses `min-h-[240px]` → this must be aligned to `h-[240px]` to prevent both the empty skeleton AND the React #418 hydration mismatch.

### Pitfall 2: Contrast violation on the payoff curve stroke
**What goes wrong:** axe-core flags the SVG curve stroke as a color-contrast failure at WCAG 1.4.3 (4.5:1 for UI elements smaller than 18.66px).
**Why it happens:** `--accent-default: oklch(0.6 0.08 70)` = `#a87c3a` = 3.44:1 on `--bg-canvas` (oklch 0.97). Sub-AA.
**How to avoid:** Use `--accent-text: oklch(0.48 0.08 70)` for the SVG stroke. This token was designed precisely for AA-safe ochre on small text; it applies equally to thin SVG strokes.
**Warning signs:** axe-core reports "Elements must have sufficient color contrast" on `.recharts-curve` or `.recharts-surface`.
**Verification:** After fix, run `pnpm test:a11y` or use axe browser extension — expect the curve stroke to clear 4.5:1.

### Pitfall 3: TypeScript errors at `strike`/`slope` access sites after union migration
**What goes wrong:** `tsc --noEmit` fails with "Property 'strike' does not exist on type 'SimulatedInstrument'".
**Why it happens:** The three consumer files (`page.tsx`, `InstrumentParams.tsx`, `InstrumentJsonLd.tsx`) currently access `instrument.strike` etc. without narrowing. After the union migration, these become type errors.
**How to avoid:** All three files must be updated atomically with `instruments.ts` in Wave 1 (data layer freeze). The pattern is `instrument.kind === 'live' ? instrument.strike : fallback`.
**Warning signs:** Pre-commit hook (`tsc --noEmit`) fails on any of the three files.
**Key constraint:** The `generateMetadata` function in `page.tsx` only accesses `instrument.id`, `instrument.name`, `instrument.nameEn`, `instrument.chainId` — all fields shared between both variants. The `generateMetadata` call site is SAFE after migration without changes.

### Pitfall 4: React #418 hydration mismatch from skeleton class divergence
**What goes wrong:** Console emits React error #418 ("Hydration failed because the server rendered HTML didn't match the client") on the instrument detail route.
**Why it happens:** `PayoffDiagramClient.tsx` loading skeleton uses `min-h-[240px]` during SSR. When recharts hydrates, the mounted `PayoffDiagram` renders a `h-[240px]` parent. React compares SSR and CSR class strings → mismatch → #418.
**How to avoid:** The `loading:` prop in `PayoffDiagramClient` must use the same class as the mounted component's outer div: `h-[240px] sm:h-[320px]`.

### Pitfall 5: `aggregateAllChains` called with a simulated instrument
**What goes wrong:** The detail page calls `aggregateAllChains()` unconditionally, then `getInstrumentPoolState(results, instrument.id, instrument.chainId)`. For a simulated entry, there is no deployed contract → the BFF/aggregator returns null/empty data (not a crash), but calling it wastes a multicall round-trip and may produce misleading log entries.
**Why it happens:** The current `page.tsx` calls `aggregateAllChains()` before any `kind` check.
**How to avoid:** Add the `kind === 'simulated'` branch immediately after the `notFound()` check, BEFORE `aggregateAllChains()`. Simulated branch never calls the aggregator; it builds pool display from `fixture.ts` directly.

### Pitfall 6: Bigint precision loss in fixture.ts
**What goes wrong:** Storing `SEEDED_LIQUIDITY` as `number` loses precision. `1_000_000 ether` in Solidity = `1e24` which exceeds `Number.MAX_SAFE_INTEGER` (2^53 - 1 ≈ 9e15).
**Why it happens:** `1e24` > `Number.MAX_SAFE_INTEGER` → any arithmetic produces incorrect results.
**How to avoid:** Store as `bigint`-as-string: `"1000000000000000000000000"`. The spec mandates this explicitly for on-chain integer quantities. The `FixtureValue<string>` type encodes this.

### Pitfall 7: ProvenancePill non-text contrast failure (WCAG 1.4.11)
**What goes wrong:** axe-core flags the pill border or icon as insufficient non-text contrast.
**WCAG 1.4.11** requires 3:1 for UI components and graphical objects.
**Why it happens:** Using `/10` opacity on `ring-border-default` (e.g., `ring-status-pass/30`) may fall below 3:1 against `bg-canvas`.
**How to avoid:** Use solid ring values (not `/30` opacity) for the provenance pills, OR verify contrast ratios at design time. The `--status-pass: oklch(0.38 0.17 145)` tokens were verified for 4.5:1 text contrast against near-white canvas — they also clear 3:1 for non-text use. Use at full opacity for pill rings.

### Schematic convex payoff generation
```typescript
// Source: payoff.ts — new function alongside existing generatePayoffData
// Simple quadratic bowl: payoff = max(k*(price-center)^2 - premium, 0)
// k = curvature controls visual amplitude; premium shifts the floor up.
export function generateSchematicConvexPayoff(
  centerPrice: number,
  premium: number,
  curvature = 0.000005,
  points = 200,
): PayoffPoint[] {
  const lo = centerPrice * 0.3
  const hi = centerPrice * 1.7
  return Array.from({ length: points }, (_, i) => {
    const price = lo + (hi - lo) * (i / (points - 1))
    const raw = curvature * (price - centerPrice) ** 2 - premium
    return { price, payoff: Math.max(raw, 0) }
  })
}
```
