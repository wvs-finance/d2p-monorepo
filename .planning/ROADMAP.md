## M1 Primary Concern — Arrival Periodicity

M1 is the SOMI-leg structural parallel of the `abrigo-x402` pipeline's data-ingestion + panel stage (its L1–L3). Its primary job is to capture the `IAgentRequester` **arrival series** (per-event timing, ordering, completeness) robustly enough that the downstream **M2 milestone — arrival-process estimation (NHPP vs Hawkes) + the four-condition convex-dominance verdict** on the SOMI/USD cost cashflow — is itself robust. M1 runs no estimation and no convex-demand verdict; it guarantees the arrival foundation those rest on. This concern is load-bearing in two phases: **Phase 1 EVENT-01** (arrival-timing fields are first-class schema columns) and **Phase 6 STATS-01** (descriptive arrival-periodicity precursor handed to M2). The M1→M2 consumer contract is therefore "arrival-process estimation + convex-demand verdict," not merely "parametric fit."

**Arrival-ordering key (HIGH-1 fix).** Somnia produces blocks at a native cadence ~14× finer than one second (scout-measured: 316M blocks since TGE ≈ 1.2M blocks/day ≈ **~72 ms/block** median; docs cite a ~20 ms consensus tick). At that cadence many EVM chains still expose `block.timestamp` only in **whole seconds**, so up to ~14 events can share a single `block_ts_utc`. The true arrival-ordering key is therefore **`(block_number, log_index)`**, with `block_ts_utc` carried as a coarse secondary timestamp. DATA-SOURCE-01 records (capability-matrix rows) both the measured median inter-block time and whether Somnia's JSON-RPC `block.timestamp` is whole-second or sub-second; EVENT-01 and the Phase 6 arrival series carry ordering on `(block_number, log_index)`. This materially shapes M2's arrival-process estimation.

## Critical Path & Parallelization

### Critical path (longest dependency chain)

```
Phase 1 → Phase 2 → Phase 3 → Phase 4 (4a-validate) → Phase 5 → Phase 6
  │                        ↘                      ↗
  │ DATA-SOURCE-01 verdict   4a-pre (concurrent with Phase 3)
  │ GATES Phase 3 INDEX-01   4b (depends on Phase 1 only)
  └──────────────────────▶   4c (depends on Phase 1 only)
```

### Parallelization summary (config.json: `parallelization: true`)

- **DATA-SOURCE-01 gates Phase 3 INDEX-01** (2026-05-29 addition): INDEX-01 cannot be authored until the free-vs-paid verdict lands in Phase 1. DATA-SOURCE-01 runs early-in-Phase-1 alongside the chain-capability probes (it absorbs the RPC-capability probe). The **schema-foundations** portion of Phase 1 does not extend the critical-path tail; the **sufficiency-verdict** portion may require a live Ormi backfill-rate probe (HIGH-4) whose duration is not doc-bounded and is the part most likely to consume the Phase 1 1.5–2× sizing allowance (LOW wall-clock-honesty fix). The provisional verdict gates Phase 3 authoring; the binding completeness re-confirmation happens inside Phase 3 (SC#6/SC#7).
- **Phase 4a is split across the Phase 3 boundary** (HIGH-1 split from prior revision):
  - **4a-pre** (Tier-B decompilation) runs **concurrently with Phase 3** — no Phase 3 dependency. Output: candidate rebate equation forms.
  - **4a-validate** (Tier-C residual + Tier-B holdout) **strictly depends on Phase 3 INDEX-01** — Tier-C requires indexed event rows.
- **4b (GAS-01)** can start as soon as Phase 1 closes (no Phase 2 or Phase 3 dependency).
- **4c (FX-01)** can start as soon as Phase 1 KPD-19 verdict lands (no Phase 2 or Phase 3 dependency).
- **Phase 3** itself admits internal parallelism (MEDIUM-2 slack): manifest scaffold + admin-event handlers (standardized EIP-1967 topic0s) can run concurrently with Phase 2's resolver work; only business-event field mappings strictly wait on the Phase 2 resolver.
- Phase 5 waits for Phase 3 AND all three Phase 4 sub-streams to land.

### Wall-clock implication

Phase 4a-pre running concurrently with Phase 3 recovers ~1× Phase 3 duration from the critical path (Tier-B decompilation is no longer serialized after indexing). Net critical-path duration is approximately `T(P1) + max(T(P2), T(P1)→T(P3) slack) + T(P3) + T(4a-validate) + T(P5) + T(P6)`, with `T(4a-pre), T(4b), T(4c)` absorbed into the `T(P3)`/`T(P4-overlap)` envelope. DATA-SOURCE-01's schema-coupled work is absorbed into the Phase 1 envelope; its live-backfill-rate-probe portion (HIGH-4) is the unbounded-by-doc-read tail most likely to consume the Phase 1 1.5–2× allowance. The DATA-SOURCE-01 verdict gates only the Phase 3 INDEX-01 authoring step.

### STATS-01 re-entry rule on observed `Upgraded` events (MEDIUM-4)

If Phase 3 INDEX-01 surfaces ≥1 `Upgraded` event whose new impl's `(impl, topic0)` field-layout-hash is not already in the Phase 2 resolver, the workflow **re-enters Phase 2** (resolver update against the new impl) **and re-runs Phase 4a-validate** against the new impl's holdout, **before** Phase 6 STATS-01 is considered valid for blocks `≥ upgrade_block`. STATS-01 reports per-impl-segment distributions; segments not fully re-validated carry `decode_status = 'unresolved_impl_transition'` and are excluded from headline numbers. This re-entry is conditional on data (no `Upgraded` event observed → no re-entry; PROJECT.md KPD-07's single-segment default applies).
