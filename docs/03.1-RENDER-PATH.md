## 4. Velite Auto-GFM Suppression (`gfm: false`)

Verified against `node_modules/velite/dist/index.js` L4982–L4989:
```
enableGfm = options.gfm ?? mdx.gfm ?? true
if (enableGfm) remarkPlugins.push(remarkGfm)
// THEN: remarkPlugins.push(...options.remarkPlugins)
```

Without `gfm: false`, the effective remark order would be:
`[remarkGfm(auto), remarkRemoveComments, remarkCopyLinkedFiles, remarkMath, remarkGfm(dup), remarkDirective]`

With `gfm: false` set on `s.mdx({...})`, the effective order is:
`[remarkRemoveComments, remarkCopyLinkedFiles, remarkMath, remarkGfm, remarkDirective, remarkTheoremDirective]`
— math first, GFM registered exactly once. Spec §0 order honored.

## 5. Locale-Derivation Mechanism

`s.path()` in Velite returns the SOURCE FILE PATH relative to the `content/` root,
e.g. `research/spike-katex.es.mdx`. The derived `locale` field uses a `.transform()`:

```ts
locale: s.path().transform((p: string) => p.endsWith('.es.mdx') ? 'es' : 'en')
```

The `.es.mdx` / `.en.mdx` suffix is present in the `s.path()` value — confirmed by
velite@0.3.1 source (`meta.path` is the source file path). This is the correct mechanism.

## 6. Components-Prop Injection Finding

**Finding:** Velite@0.3.1 compiles MDX bodies with `outputFormat: 'function-body'` and
**passes NO `providerImportSource`**. The compiled default export resolves custom components
(JSX elements whose names start with uppercase letters, e.g. `<Figure>`, `<TheoremBlock>`)
via the `components` PROP passed to the evaluated function — NOT via `useMDXComponents()`.

A no-prop renderer (`return <C />`) therefore **throws at runtime** on any MDX body containing
custom components. The extended `MDXRenderer` passes the components map explicitly:
`return <C components={components} />`.

This is the critical path Plan C depends on: the component-map (h2/h3, Figure, TheoremBlock,
Sidenote) is passed as a prop to the compiled body's default export at render time.
