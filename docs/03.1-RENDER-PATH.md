## 4. Velite Auto-GFM Suppression (`gfm: false`)

Verified against `node_modules/velite/dist/index.js` L4982–L4989:
```
enableGfm = options.gfm ?? mdx.gfm ?? true
if (enableGfm) remarkPlugins.push(remarkGfm)
// THEN: remarkPlugins.push(...options.remarkPlugins)
```

Without `gfm: false`, the effective remark order would be:
`[remarkGfm(auto), remarkRemoveComments, remarkCopyLinkedFiles, remarkMath, remarkGfm(dup), remarkDirective]`

With `gfm: false` set on `s.mdx({...})`, the effective order is:
`[remarkRemoveComments, remarkCopyLinkedFiles, remarkMath, remarkGfm, remarkDirective, remarkTheoremDirective]`
— math first, GFM registered exactly once. Spec §0 order honored.

## 5. Locale-Derivation Mechanism

`s.path()` in Velite returns the SOURCE FILE PATH relative to the `content/` root,
e.g. `research/spike-katex.es.mdx`. The derived `locale` field uses a `.transform()`:

```ts
locale: s.path().transform((p: string) => p.endsWith('.es.mdx') ? 'es' : 'en')
```

The `.es.mdx` / `.en.mdx` suffix is present in the `s.path()` value — confirmed by
velite@0.3.1 source (`meta.path` is the source file path). This is the correct mechanism.

## 6. Components-Prop Injection Finding

**Finding:** Velite@0.3.1 compiles MDX bodies with `outputFormat: 'function-body'` and
**passes NO `providerImportSource`**. The compiled default export resolves custom components
(JSX elements whose names start with uppercase letters, e.g. `<Figure>`, `<TheoremBlock>`)
via the `components` PROP passed to the evaluated function — NOT via `useMDXComponents()`.

A no-prop renderer (`return <C />`) therefore **throws at runtime** on any MDX body containing
custom components. The extended `MDXRenderer` passes the components map explicitly:
`return <C components={components} />`.

This is the critical path Plan C depends on: the component-map (h2/h3, Figure, TheoremBlock,
Sidenote) is passed as a prop to the compiled body's default export at render time.

## 7. next-mdx-remote/rsc Fallback Exercise

**Verdict: PASS** (exercised 2026-05-29)

The fallback path was exercised by running `compileMDX` from `next-mdx-remote/rsc.js`
directly against the `spike-katex.es.mdx` source with the same remark-math + rehype-katex
plugin chain:

```js
const result = await compileMDX({
  source,
  options: {
    parseFrontmatter: true,
    mdxOptions: {
      remarkPlugins: [remarkMath],
      rehypePlugins: [[rehypeKatex, { output: 'htmlAndMathml', throwOnError: false }]],
    },
  },
})
```

Result: `frontmatter.slug = 'spike-katex'`, `typeof content = 'object'` (React element). **PASS.**

**Finding:** The `next-mdx-remote/rsc` path compiles MDX at render time (pulling `@mdx-js/mdx`
into the render path) and returns a React element directly (unlike `s.mdx()` which returns a
function-body string for evaluation). The render paths are fundamentally different:
- **Primary path (s.mdx + MDXRenderer):** compile at Velite build time → store string → evaluate at render. Zero runtime MDX cost. Best for 3G LCP.
- **Fallback path (next-mdx-remote/rsc):** compile at request time → React element. Higher per-request cost; only used if s.mdx() fails.

The primary path (`s.mdx()` + MDXRenderer) is confirmed working; the fallback is confirmed
as a viable emergency option but is NOT used in the v1 implementation.

## 10. v1 LCP Font Strategy (Plan C2, 03.1-04) — `font-display: swap` + stable-path preload

**Date:** 2026-05-29 · **Plan:** 03.1-04 (Plan C2 perf gate)

### The problem (recap of the Wave 1 burn)

`katex/dist/katex.min.css` declares all ~17 font faces with `font-display: block` (FOIT —
invisible text until the face loads, the worst LCP behavior for math-heavy pages). Two facts
made the naive fix fail:

1. **Turbopack rewrites the `@font-face` `src` URLs** in `katex.min.css` (`url(fonts/…woff2)`)
   to hashed `/_next/static/media/<hash>.woff2` paths that are NOT knowable at author time —
   so you cannot author a `<link rel=preload>` against them.
2. `katex.min.css` is **vendored in `node_modules`** — its `@font-face` `font-display: block`
   cannot be edited in place.

Wave 1 commit `4117fc3` removed an earlier preload that pointed at `/static/katex/*.woff2`
**which did not exist on disk** → 4×404 + 2 console errors, and the swap `@font-face` was inert
because the referenced asset was absent.

### The v1 fix (applied here — NO Python tooling)

For the **2 LCP-critical faces only** (`KaTeX_Main-Regular`, `KaTeX_Math-Italic` — the
upright text and math-italic glyphs that dominate the rendered-math first paint):

1. **Copy the 2 woff2 files to `public/static/katex/`.** Files under `public/` are served by
   Next at a **stable, non-rewritten** `/static/katex/<name>.woff2` path. (The copy is a
   committed build asset — `public/static/katex/KaTeX_Main-Regular.woff2` +
   `KaTeX_Math-Italic.woff2`, ~42 KB combined.)
2. **`<link rel="preload" as="font" type="font/woff2" crossorigin>`** the two stable paths in
   `app/(lab)/research/layout.tsx`. Because the path is stable (not Turbopack-hashed) the
   preload resolves **200, not 404**.
3. **Route-scoped `@font-face` override** in the same layout, declared AFTER the
   `import 'katex/dist/katex.min.css'`, re-declaring `KaTeX_Main` (normal/400) and
   `KaTeX_Math` (italic/400) with `font-display: swap` and `src` pointing at the stable
   `/static/katex/*` paths. Because it uses the same family names and is declared later, it
   wins over the bundled `block` rule for these two families. The rendered math binds to the
   swap-enabled, preloaded faces.

The remaining ~15 KaTeX faces (script, fraktur, AMS, sizes, caligraphic, etc.) keep the
bundled `@font-face`/`block` default — they are off the LCP critical path and only appear in
rarer notation.

### `unicode-range` fallback (spec §5)

KaTeX's own `@font-face` declarations carry **no `unicode-range`** — each face loads the whole
file (KaTeX maps glyphs to faces via per-glyph CSS classes, not the browser's unicode-range
matching). So the v1 strategy deliberately does **not** add `unicode-range` to the two override
faces (adding one would risk excluding glyphs KaTeX expects from the full face). The documented
fallback if the LCP gate still fails on 3G is: serve the system math fallback during the swap
period (already implicit with `font-display: swap` — the browser renders fallback glyphs
immediately, then swaps to KaTeX_Main/KaTeX_Math when they arrive), and, only if still failing,
escalate to the v2 byte-subset below.

### DEFERRED to v2 — byte-level subsetting

`fonttools` / `pyftsubset` is **Python tooling and is NOT installed**; provisioning a Python
font-build step is explicitly **out of scope for v1** and **DEFERRED to v2**. v2 would subset
`KaTeX_Main-Regular` + `KaTeX_Math-Italic` to the actually-used glyph set (smaller download,
faster swap), and may add a proper `unicode-range`-partitioned face set. v1 ships the full
faces with `swap` + stable-path preload — the safe, no-Python LCP baseline.
